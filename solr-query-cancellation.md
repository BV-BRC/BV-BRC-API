# Solr Query Cancellation for Abandoned Browser Requests

## Background

Broad crossCollection join queries (e.g. `taxon_lineage_ids:2` = all Bacteria, hitting
93M docs/shard) cause JVM heap OOM on Solr data nodes. The 2026-06-25 crash killed three
nodes. A key contributing factor: browser users submit a slow join query then navigate
away, but the Solr query runs to completion (or OOM) regardless.

Cluster: SolrCloud 9.6.1. Coordinator nodes: walnut, chestnut. Default URL:
`https://bio-gp1.cels.anl.gov:15183/solr`

## Why Solr Cannot Self-Detect Abandonment

Request chain:

```
Browser → plum-web/grape-web → BV-BRC API (p3api) → Solr coordinator → data nodes
```

When the browser navigates away, the browser closes its connection to the web app. By
then p3api has already dispatched a new, independent HTTP request to Solr. The Solr query
runs on a Jetty thread that is not interrupted by upstream disconnection. The thread blocks
in DocSet construction until done (or OOM).

## Solution: Solr 9.x Task Management API

Solr 9.6.1 has full task cancellation support. Two parts are needed:

### 1. Tag Solr requests as cancellable (p3api change)

Add to every Solr query dispatched for expensive operations (genome_feature joins at
minimum):

```
canCancel=true
queryUUID=<uuid>    # generated by p3api per request
```

Without `canCancel=true` the query is not tracked and cannot be cancelled.

### 2. Cancel on browser disconnect (p3api change)

Node.js example:

```javascript
const crypto = require('crypto');

async function solrQuery(req, res, solrParams) {
    const queryUUID = crypto.randomUUID();

    solrParams.canCancel  = 'true';
    solrParams.queryUUID  = queryUUID;

    req.on('close', async () => {
        if (!res.headersSent) {
            // Browser disconnected before response was sent — cancel in-flight Solr query
            try {
                await fetch(
                    `${SOLR_URL}/admin/tasks/cancel?queryUUID=${queryUUID}`
                );
            } catch (e) {
                // best-effort; log and continue
            }
        }
    });

    // ... dispatch solrParams to Solr as normal ...
}
```

### Task Management API endpoints

```bash
# List active cancellable tasks (run on coordinator)
GET /solr/admin/tasks/activetasks

# Cancel a specific task
GET /solr/admin/tasks/cancel?queryUUID=<uuid>
```

Credentials via .netrc or basic auth as used elsewhere in the cluster.

## Caveats

- **Data nodes are not directly cancelled.** The cancel signal reaches the coordinator;
  data nodes may continue their shard sub-queries briefly before the coordinator stops
  waiting. DocSets already allocated on data nodes are not immediately freed.

- **canCancel overhead.** Solr tracks cancellable tasks in memory. Acceptable for
  expensive join queries; not needed for fast lookups. Scope it to genome_feature queries
  or any query with a crossCollection join in the fq.

- **req.on('close') fires on both disconnect AND normal completion.** Guard with
  `!res.headersSent` (shown above) so you don't spuriously cancel queries that finished
  successfully.

## Belt-and-Suspenders: timeAllowed

Add `timeAllowed=60000` (60 seconds) to all genome_feature queries regardless of the
cancel implementation. This is a Solr-native hard limit requiring no connection tracking.
If p3api crashes or a disconnect goes undetected, timeAllowed still bounds the damage.
The Coronaviridae queries ran for 330 seconds; a 60s limit cuts their heap residency
by ~82%.

## Recommended Priority

1. **timeAllowed=60000** on genome_feature queries — one line, immediate protection,
   no coordination required.
2. **canCancel + queryUUID** on join queries — enables cancellation infrastructure.
3. **req.on('close') cancel call** — completes the abandonment detection loop.
4. **CircuitBreakerPlugin** on Solr side — rejects new queries when heap > threshold,
   prevents pile-ons when cluster is already under pressure.
